# June 2026 Target

Status: canonical target, not a current-state claim. The production matrix tracks the unproven parts until browser, backend, eval, and media gates pass.

## Thesis

NodeRoom is a finance-grade collaborative workroom where humans and agents safely produce auditable Excel-like work products: models, proofs, memos, charts, citations, traces, and exports.

The durable product mental model:

> Binder tells you what exists. Stage shows the work. Copilot drives the work. Status proves what just happened.

The durable agent mental model:

> LLM routes intent. Harness schedules work. Ledger commits truth.

## Desktop Shell Contract

The target desktop shell is four roles, not four equal cockpits:

| Surface | Owns | Must not own |
|---|---|---|
| Room Binder / Deal Binder | Source files, workbooks, work products, people, agents, review/proof, permissions | Full agent streams, queues, logs, protocol labels, or low-level tool controls |
| Work Surface | Spreadsheet, proof, source document, memo, chart, trace artifact, gold comparison | Permanent chat dominance or unrelated room navigation |
| Copilot | Chat, work queue, agent steering, active agent streams, artifact cards, task controls | Canonical spreadsheet state or hidden durable writes |
| Bottom Signal Tape + Status Strip | Ambient events plus short authoritative truth for commits, conflicts, evals, sync, cost, and runtime | Large proof/source drawers or permanent competing work panes |

The target can use three columns on desktop only when the left column is a stable binder, the center is the primary work surface, and the right side is Copilot/orchestration.

## Binder Contract

The binder is navigational. It answers "what exists in this room?"

- Source Files
- Workbooks
- Work Products
- People
- Agents
- Review & Proof
- Permissions

Click behavior:

- Clicking an artifact opens it on the center stage.
- Clicking a person jumps to their current location or visible work context.
- Clicking an agent highlights the claimed range and opens Copilot detail.
- Clicking a privacy or policy item opens the proof/policy artifact.

Anti-patterns:

- Do not turn the binder into a second agent cockpit.
- Do not put full streams, logs, queues, or protocol implementation labels in the binder.
- Do not make the binder compete with the work product for attention.

## Work Surface Contract

The center is where professional work is inspected and edited. Sources, proofs, traces, and gold comparisons open on the stage or as a center split within the stage, not as a new global panel.

For finance and GTM workflows, the stage must preserve spreadsheet muscle memory:

- keyboard-first navigation and edit modes,
- formula/value distinction,
- evidence overlays that do not break the grid,
- source/proof panes that can be opened just-in-time,
- no-clobber conflict feedback next to the affected work.

## Copilot Contract

Copilot owns conversation and operation steering:

- public and private chat modes,
- work queue,
- active agent streams,
- agent cards,
- task controls,
- plan previews,
- cancellations, retries, priorities, and clarification.

The agent may recommend command-like intent. The harness validates, schedules, and commits safe mutation-like writes. Selection, viewport, cursor, and presence remain ephemeral operation-like state.

## Bottom Contract

The bottom is not a drawer by default.

- Signal Tape is ambient and configurable. It can show source updates, agent events, risk, eval, cost, or collaboration activity.
- Status Strip is authoritative and short. It tells what just happened: committed, conflict, draft, synced, failed, retried, budgeted, or paused.
- Detailed proof/source/gold/trace views open on the center stage.

## Responsive Contract

| Width | Target behavior |
|---|---|
| `>=1440px` | Full binder, center work surface, right Copilot, bottom status |
| `1200-1439px` | Narrow binder or icon rail with peek behavior |
| `900-1199px` | Binder behind a Room button; center and Copilot remain usable |

## Agent And Scheduler Invariants

- Every user message becomes an intake event: new command, steering patch, parallel subagent, wait, clarification, note, cancel, or priority change.
- Expensive work gets a preflight PlanPreview: target artifacts, read/write/affected sets, dependency expansion, conflicts, scheduling recommendation, cost, and runtime.
- Affected sets include formulas, charts, memos/wiki refs, source refs, active human edits, proposals/drafts, agent claims, and privacy boundaries.
- Agents propose commands. The harness turns safe parts into mutations. Ephemeral collaboration stays as operations.
- Soft intent claims and short commit leases protect collaboration without blocking human typing.
- The LLM must not directly schedule, lock, interrupt, or write durable cells.
- Managed writes own lock, CAS, proposal, review, and trace behavior.
- Chat claims are not evidence. Durable evidence must point to source artifacts, cells, pages, rows, boxes, traces, or validator output.
- Formula cells must not be overwritten with scalar text unless the operation is explicitly a formula replacement.

## Workbook Runtime Target

Long term, the spreadsheet surface needs a real workbook runtime rather than a hand-built table:

- workbook runtime for local edit semantics, formulas, selection, fill, and overlays,
- Convex ledger for canonical room truth, CAS, locks, proposals, traces, permissions, and sync,
- NodeAgent harness for context, scheduling, provider calls, evidence, and commits,
- headless/worker validation for formulas and export/reopen checks.

This is an adapter target, not a commitment to one runtime until the POC passes.

## Evaluation Target

NodeRoomBench should score the product as a harnessed collaborative work system, not as a raw model.

Evaluate:

- final workbook correctness,
- formula correctness,
- formatting and usability,
- source citations and evidence,
- no-clobber behavior,
- multi-agent scheduling and affected-set planning,
- privacy boundaries,
- cost and runtime,
- trace quality,
- final artifact usefulness to a finance/GTM professional outside NodeRoom.

Official benchmark mode is stricter than NodeRoomBench. BankerToolBench and
SpreadsheetBench readiness must be tracked by the official benchmark gate, not
inferred from internal professional workflow evals. That gate has to prove
official task ingest, hidden-gold isolation, runner adapters, trajectory/cost
capture, workbook export/reopen fidelity, and benchmark-native scoring before
the README can claim benchmark readiness.

Allowed and required reporting:

- model,
- harness,
- tool policy,
- budget,
- evaluator,
- retries/failures,
- route,
- final artifact,
- trajectory.

Disallowed:

- hidden gold access,
- benchmark-specific hardcoding,
- evaluator or test mutation,
- public answer lookup,
- tuning on held-out cases,
- hiding retries or failures,
- reporting raw model capability when the harness did the reliability work.

- the shell now uses the target surface roles: Room Binder, center Work Surface, right Copilot, and shell-level Signal Tape/Status Strip;
- center-stage split mode exists in memory mode so a secondary proof/source/wiki surface can sit beside the primary work surface without moving into Copilot or the bottom strip;
- public/private chat lanes now live in Copilot tabs;
- resizable panels exist;
- public/private chat and artifact surfaces exist;
- deterministic `IntakeDecision` and `PlanPreview` contracts classify messages, expand affected sets, and block privacy/formula/budget unsafe plans before provider spend;
- trace/proposals/auto-accept exist;
- managed lock/CAS/draft behavior and model ladder evals exist;
- public-gold, finance, professional workflow, and QA matrix infrastructure exist.
- official benchmark readiness reporting now exists and is wired into the agent
  improvement loop / HALO deterministic lane.
- BankerToolBench official local-bundle ingest exists for already-downloaded
  `tasks.jsonl` / `task-data` / `golden-outputs` bundles. It parses input files,
  workflow taxonomy, and weighted rubric metadata while keeping rubric, canary,
  and golden-output paths out of the agent-facing task payload.
- BankerToolBench sandbox staging exists: each task gets an `agent/` manifest
  with only the official default `final_prompt` and input files, plus a separate
  `evaluator/` manifest with prompt context, formatting context, canary,
  weighted rubric, and golden outputs. A local BTB-shaped smoke fixture verifies
  zero agent-side golden/rubric/canary leaks.
- BankerToolBench local runner plumbing exists: `npm run
  benchmark:bankertoolbench:run` emits candidate deliverables from an
  agent-only workspace before opening evaluator-only weighted rubric and golden
  files. The local exact-golden verifier smoke proves the runner/scorer boundary
  and trace shape, not official Harbor/Gandalf scoring.
- SpreadsheetBench V1/V2 official local-bundle ingest exists, with agent-facing
  inputs/prompts separated from evaluator-only golden workbooks and scorer
  metadata. Smoke artifacts cover the V1 verified-400 bundle and V2 public
  example bundle.
- SpreadsheetBench sandbox staging exists: each task gets an `agent/` manifest
  with only instructions, prompts, and input workbooks, plus a separate
  `evaluator/` manifest with golden files and scorer metadata. Smoke artifacts
  verify zero agent-side gold/scorer leaks for V1/V2 samples.
- SpreadsheetBench workbook scoring exists for local candidate-vs-gold baselines:
  values, formulas, and optional stable style fingerprints can be compared after
  reopening `.xlsx` files. Smoke artifacts compare official input workbooks
  against evaluator-only gold to prove the diff path; they are not model scores.
- SpreadsheetBench staged runner plumbing exists: a copy-input baseline reads
  only the staged `agent/` manifest, emits a candidate workbook, then opens the
  `evaluator/` manifest for scoring. Smoke artifacts record trajectory, timing,
  model-call count, provider cost, and failed baseline scores. This is runner
  plumbing, not a model/export worker.
- SpreadsheetBench edit/export plumbing exists: `--mode apply-agent-patch` reads
  `agent/edit-plan.json`, applies cell-level value/formula/style operations,
  writes a candidate workbook, then opens the evaluator manifest for scoring.
  A smoke artifact records a passing edited workbook with zero mismatches. This
  proves workbook edit/export/reopen packaging; it is still not a model score.
- SpreadsheetBench model-edit plumbing exists: `--mode model-edit-plan --model
  <route>` snapshots only the staged `agent/` workbook and prompts, asks the
  configured model for a JSON edit plan, applies the plan, records token/cost
  usage, emits a candidate workbook, then opens the evaluator manifest for
  scoring. A one-task live smoke with `gpt-5.4-nano` passed and recorded model
  call, trajectory, timing, and cost. This is still not an official benchmark
  score until it runs against official bundles under the documented policy.
- SpreadsheetBench runner attempts now create an explicit `agent-workspace/`
  before candidate generation. Candidate generation reads copied agent-visible
  manifests, workbooks, prompts, and edit plans from that workspace; each retry
  records a workspace manifest in the trajectory, and contamination scans cover
  those manifests. This proves the runner's file-view boundary; it is not yet
  OS/Docker process isolation.
- SpreadsheetBench model-edit snapshots now use a larger bounded workbook
  context so medium official workbooks are not silently truncated, and the
  runner caches simple `SUM(...)` formula results during export/reopen. This
  improves harness fidelity without adding answer lookup.
- SpreadsheetBench official V1 failed-path accounting now exists: a live
  `model-edit-plan` run against a staged V1 task recorded that `gpt-5.4-nano`
  chose a missing sheet name, then counted the task as failed with model call,
  token usage, cost, trajectory, and error instead of hiding the spend/failure
  in warnings. This proves path accounting, not task success.
- SpreadsheetBench repeated-run accounting now exists: `--repeats 5` records
  case count, attempt count, pass rate, p50/p95/max latency, failure taxonomy,
  model calls, tokens, and provider cost. The checked-in official V1 N=5 smoke
  recorded 5 attempts on one task, 0/5 pass, p95 4.462s, $0.0078935 spend,
  three invalid-sheet candidate-generation failures, and two scored partial
  candidates. This proves drift visibility, not benchmark readiness.
- SpreadsheetBench retry-policy accounting now exists: `--retry-failed N`
  retries candidate-generation and scoring errors, `--retry-score-failures`
  explicitly opts into retrying scored-but-wrong candidates, and the report
  records case-level attempts, retry exhaustion, pass-after-retry counts,
  latency, token use, and provider cost. The checked-in official V1 retry smoke
  ran one task with `gpt-5.4-nano`, `--retry-failed 2`, and
  `--retry-score-failures`: all 3 attempts reached scoring, best overall was
  0.6, p95 latency was 7.609s, spend was $0.00661755, and pass remained 0/3.
  This proves auditable retry behavior and the current planner gap at the same
  time.
- Official benchmark contamination checking now exists: `npm run
  benchmark:contamination` scans agent-facing task manifests, candidate
  manifests, and generated edit plans for evaluator-only gold/rubric/canary
  metadata. Checked-in smokes record 0 leaks for the staged SpreadsheetBench V1
  root, the N=5 V1 candidate output, the retry V1 candidate output, and the
  staged BankerToolBench fixture.
  This proves metadata-boundary hygiene, not Docker/process isolation.
